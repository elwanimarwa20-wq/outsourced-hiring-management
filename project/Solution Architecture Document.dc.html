1 · Executive Summary & Platform Recommendation

The solution is a centralized, bilingual (English / Arabic) hiring-form management application. It replaces fragmented spreadsheets and manual approval chains with a single auditable system of record covering the full lifecycle of an outsourced or service-contract hire — from form creation through HR review, Sector Head approval, completion, and reporting.

✓ Recommended: Oracle APEX

Low-code, runs natively on the Oracle Database, fastest path to a custom forms-and-approval app, fully responsive, built-in auth/roles, and zero additional middleware. Ideal for a focused departmental application like this.

Oracle Fusion HCM

Used as the integration source for employee/position master data — not as the build platform. Fusion's recruiting module is heavier than needed for this custom outsourced-staff form and is harder to tailor to the exact 4-section layout and cost logic.

Verdict: Build on Oracle APEX over the Oracle Database; integrate read-only with Oracle HCM via REST for employee and position lookups. This balances speed of delivery, low cost of ownership, and full control over the bilingual UI.

2 · Solution Architecture

A standard three-tier Oracle APEX topology. The browser talks to Oracle REST Data Services (ORDS); APEX renders pages and runs PL/SQL business logic directly against the Oracle Database; integrations and document storage sit alongside.

  • Audit trail & version history via row-level triggers writing to HR_FORM_AUDIT.
  • Role-based access control using APEX authorization schemes mapped to three roles.
  • Document repository for signed forms stored as BLOBs (or OCI Object Storage for scale).
  • HCM integration read-only REST calls to populate employee & position master data.

3 · Database Schema & ERD

Five core objects in schema HR_OSF. HR_HIRING_FORMS is the master entity; attachments and audit rows are children keyed by FORM_ID. Lookups are reference tables; users/roles align to APEX security.

4 · SQL Tables & DDL Scripts

Annual salary and total cost are virtual (computed) columns so the database guarantees the arithmetic — no risk of stale stored values.

-- Master: hiring forms
CREATE TABLE hr_hiring_forms (
  form_id           VARCHAR2(20)  PRIMARY KEY,
  employee_name     VARCHAR2(150) NOT NULL,
  employee_name_ar  VARCHAR2(150),
  dob               DATE,
  nationality       VARCHAR2(60),
  passport_no       VARCHAR2(30),
  email             VARCHAR2(120),
  mobile            VARCHAR2(30),
  position          VARCHAR2(120),
  contract_type     VARCHAR2(30) CHECK (contract_type IN ('OUTSOURCED','SERVICE_CONTRACT')),
  start_date        DATE,
  prev_employer     VARCHAR2(150),
  prev_monthly      NUMBER(12,2),
  prev_benefits     VARCHAR2(300),
  prev_annual       NUMBER(12,2),
  proposed_monthly  NUMBER(12,2) NOT NULL,
  -- auto-calculated, always correct:
  annual_salary     NUMBER(14,2) GENERATED ALWAYS AS (proposed_monthly * 12) VIRTUAL,
  medical_cost      NUMBER(12,2) DEFAULT 0,
  ticket_cost       NUMBER(12,2) DEFAULT 0,
  visa_cost         NUMBER(12,2) DEFAULT 0,
  total_cost        NUMBER(14,2) GENERATED ALWAYS AS
                      (proposed_monthly*12 + medical_cost + ticket_cost + visa_cost) VIRTUAL,
  status            VARCHAR2(20) DEFAULT 'DRAFT' CHECK (status IN ('DRAFT','COMPLETED')),
  hiring_manager    VARCHAR2(120),
  created_by        VARCHAR2(120),
  created_dt        DATE DEFAULT SYSDATE,
  updated_dt        DATE DEFAULT SYSDATE
);

-- Child: attachments (with version history)
CREATE TABLE hr_form_attachments (
  attach_id    NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  form_id      VARCHAR2(20) REFERENCES hr_hiring_forms(form_id),
  file_name    VARCHAR2(255),
  mime_type    VARCHAR2(120),
  file_blob    BLOB,
  file_size    NUMBER,
  version_no   NUMBER DEFAULT 1,
  uploaded_by  VARCHAR2(120),
  uploaded_dt  DATE DEFAULT SYSDATE
);

-- Child: audit trail
CREATE TABLE hr_form_audit (
  audit_id   NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  form_id    VARCHAR2(20) REFERENCES hr_hiring_forms(form_id),
  action     VARCHAR2(80),
  old_value  CLOB,
  new_value  CLOB,
  acted_by   VARCHAR2(120),
  acted_dt   TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- Audit trigger
CREATE OR REPLACE TRIGGER trg_hf_audit
AFTER UPDATE ON hr_hiring_forms FOR EACH ROW
BEGIN
  INSERT INTO hr_form_audit (form_id, action, old_value, new_value, acted_by)
  VALUES (:new.form_id, 'UPDATE', :old.status, :new.status, v('APP_USER'));
END;

5 · Business Logic & Validation Rules

Rule Behaviour
Annual Salary= Proposed Monthly × 12 — computed, read-only.
Total Employment Cost= Annual Salary + Medical + Air Ticket + Visa — computed, read-only.
Editable in all statusesDraft and Completed forms remain fully editable; every change is audited.
Required fieldsName, Nationality, Position, Contract Type, Proposed Monthly Salary.
ValidationEmail format, salary > 0, passport pattern, start date ≥ today (warn, not block).
Form IDAuto-generated sequence: OSF-YYYY-NNNN.

6 · Approval Workflow

The hiring form follows an HR-routed approval chain ending with Sector Head sign-off. Forms can be returned for correction at any review step; all transitions are logged to the audit trail.

9 · Reporting & Analytics

  • Total hiring forms · Completed vs Draft breakdown.
  • Monthly hiring trend (12-month rolling).
  • Total projected employment cost (AED) across all forms.
  • Average proposed salary by contract type.
  • Exportable to Excel / PDF; KPI tiles on the dashboard.

10 · PDF Print Layout

A branded, A4 print template renders all four sections, the auto-calculated totals, and a dual signature block (Hiring Manager + Sector Head) with date lines. Generated via APEX Report Query / BI Publisher, or directly from the prototype's Print view. Output carries the Ministry masthead and a confidentiality footer.

11 · Deployment Plan

  1. Provision — DEV / TEST / PROD workspaces on Oracle APEX (or Autonomous DB).
  2. Schema build — run DDL scripts; load reference data; configure sequences.
  3. App import — deploy APEX application export; bind authentication to Ministry SSO.
  4. Integration — configure ORDS REST endpoints to Oracle HCM; test employee lookup.
  5. UAT — execute scenarios in §12 with HR & Hiring Manager users.
  6. Go-live — data migration of existing forms, cutover, hypercare (2 weeks).
